Github API

 2020-08-06
  github
  5 minutes read (About 706 words) 0 visits

For users who use Github, there are often some automation requirements. For example, my requirement is to regularly backup Github issues and comments locally. The following is a reference for using Github’s API

Basic Access Path (Root Endpoints)

When reading the document at the beginning, follow its example to curl directly on the command line, or access it in InSomnia or Postman software, and the 200 status is perfectly displayed. But once I rewrite my username in the link, it will display 404 without page. I thought it was an authorization problem, and then tried various ways in the header HEADER for username and token keys, but it was useless or 404. It turned out that it was not a problem with the method, but the link address was not written correctly!
In fact, if you just read it, you don’t need any authorization at all. You can directly enter the link in the command line, Insomnia, web page and other situations to access all public information of anyone.
Then check the official path list Root Endpoints to get the link, it seems that how to access it is wrong. Instead, I saw a link in Stackoverflow. I found various correct access paths along the way, summarized as follows:

First of all! The visited link cannot have / at the end. For example, https://api.github.com/users/solomonxie can access my personal information, but https://api.github.com/users/solomonxie/ does not work. The only difference is that there is one more/.
Second! Unlike general URL access, Github’s API access links are case sensitive!
Personal main information. https://api.github.com/users/ username, the data obtained is as follows:
image

Personally owned repo. https://api.github.com/users/username/repos. Will get a list of repo in JSON format.
Repo details. https://api.github.com/repos/username/warehouse name. The path of the repo began to be different from the personal information.
Get the content list of a repo. https://api.github.com/repos/solomonxie/gists/contents, note that this will only return the contents of the root directory.
Get the content list of subdirectories in the repo. https://api.github.com/repos/solomonxie/gists/contents/ directory name. It must be noted that the capitalization of the original file name must be completely followed here, otherwise the information cannot be obtained. If it is a deeper content, write the name of the directory in the link column in order.
Get information about a file in the repo (not including content). https://api.github.com/repos/solomonxie/gists/contents/ file path. The file path is the full path of the file and is case sensitive. Only basic file information will be returned.
Get the original content (Raw) of a file. 1. Extract the download_url link from the above file information, and then you can get its original content. 2. Or visit directly: https://raw.githubusercontent.com/username/warehouse name/branch name/file path
A list of all commits in the repo. https://api.github.com/repos/username/warehouse name/commits.
Details of a certain commit. https://api.github.com/repos/username/warehouse name/commits/SHA of a certain commit
List of issues. https://api.github.com/repos/username/repository name/issues.
Details of an issue. https://api.github.com/repos/username/warehouse name/issues/ serial number. Issues are listed in the order of 1, 2, and 3.
A list of comments in an issue. https://api.github.com/repos/username/warehouse name/issues/serial number/comments.
Details of a comment. https://api.github.com/repos/Username/Repository Name/issues/comments/ ID of comment details. The comment ID is obtained from the issues list.

Query parameters (Parameters)

If you add query conditions to the basic link above, then the returned data is filtered. For example, it is required to return only the open issues, or to display the list data in pages. Commonly used as follows:

Paging function. The format is?page=page number&per_page=each page contains number.
Such as https://api.github.com/users/solomonxie/repos?page=2&per_page=3

Issue status. The format is?state=state.
Such as https://api.github.com/repos/solomonxie/solomonxie.github.io/issues?state=closed
